# 钻井工程数字化系统 — 完整方案

## 一、系统定位与目标

**定位**：简洁实用的钻井工程数字化系统，面向陆地/单井主流作业场景。

**核心目标**：
- **轨道设计（融防碰）**：七段式井眼轨迹优化，内置邻井防碰约束，一键出设计参数与轨迹图。
- **基础数据与录井管理**：井身结构、井口坐标、邻井轨迹等基础数据与录井数据的录入、存储、查询。
- **防碰扫描**：对设计/实钻轨迹与邻井进行防碰扫描，输出最小距离、风险等级、扫描报告。

**设计原则**：简化操作流程、保障钻井设计安全高效、满足现场基础作业需求。

### 2.2 模块划分

| 模块 | 职责 | 对外接口 |
|------|------|----------|
| **轨迹设计模块** | 靶点与井口输入、算法选择、参数配置、调用优化引擎、结果展示与导出 | REST/内部 API：提交设计任务、查询结果、获取轨迹图 |
| **基础数据模块** | 井基本信息、井口坐标、井身结构、邻井轨迹文件(Excel)管理、录井数据 | CRUD API、文件上传、列表/筛选/导出 |
| **防碰扫描模块** | 给定轨迹+邻井集合，执行扫描、最小距离、风险等级、报告生成 | 单次扫描 API、批量扫描、报告下载 |
| **优化引擎** | 统一封装 PSO/B2OPT/GA-optiGAN 等，七段式目标+防碰约束 | `optimize_well_trajectory(target_e, target_n, target_d, ...)` 统一入口 |
| **可视化与报表** | 轨迹 3D 图、防碰示意图、设计/扫描报告、Excel 导出 | 图片 URL、PDF/Excel 流 |

### 2.3 数据流概览

- **轨迹设计**：用户输入靶点(E,N,D)、井口、邻井列表 → 选择算法与参数 → 后端调用优化引擎 → 返回最佳参数与轨迹 → 前端展示 3D 图与参数表，可导出报告。
- **基础数据**：用户维护井、井口、上传邻井 Excel → 存入数据库/文件存储 → 在设计与防碰中按井号/项目选用。
- **防碰扫描**：用户选择“设计井轨迹”+“邻井列表”+安全半径 → 后端执行扫描 → 返回最小距离、风险段、报告 → 前端展示结果与示意图。

## 三、核心功能与用户界面

### 3.1 功能总览

| 功能 | 简要说明 | 主界面入口 |
|------|----------|------------|
| **轨道设计（融防碰）** | 输入靶点与井口，选择优化算法，一键生成七段式设计参数与轨迹，含邻井防碰 | 「轨迹设计」 |
| **基础数据管理** | 井信息、井口坐标、邻井轨迹(Excel) 的增删改查与导入导出 | 「基础数据」 |
| **录井管理** | 录井数据上传、关联井、查询与简单曲线展示 | 「录井管理」 |
| **防碰扫描** | 对一条轨迹与多口邻井做防碰扫描，得到最小距离与风险提示 | 「防碰扫描」 |
| **报表与导出** | 设计报告、防碰报告、轨迹图、Excel 汇总 | 各功能内「导出」按钮 |

### 3.2 用户界面结构（建议）

- **顶部**：系统名称 + 主导航：「轨迹设计」「基础数据」「录井管理」「防碰扫描」。
- **轨迹设计页**  
  - 左侧：步骤条（1. 靶点与井口 2. 邻井选择 3. 算法与参数 4. 结果与导出）。  
  - 中/右：表单 + 结果区（参数表、3D 轨迹图、邻井叠加）。  
  - 操作：保存为设计井、导出报告/图、发起防碰扫描。
- **基础数据页**  
  - 子 Tab：井列表 | 井口坐标 | 邻井轨迹文件。  
  - 井列表：表格 + 筛选 + 新增/编辑/删除。  
  - 邻井轨迹：上传 Excel、关联井号、解析预览。
- **录井管理页**  
  - 按井/时间筛选；上传录井文件；表格+简单曲线（深度-参数）。
- **防碰扫描页**  
  - 选择“待扫描轨迹”（来自设计或上传）、邻井多选、安全半径 → 执行扫描 → 结果表 + 最小距离 + 风险等级 + 报告导出。

### 3.3 交互逻辑要点

- **轨迹设计**：默认带出当前项目/区块常用井口与邻井，减少重复输入；算法可选 PSO/B2OPT/GA-optiGAN，高级参数可折叠。
- **基础数据**：井与井口、邻井文件关联清晰；删除井时提示是否影响已有设计与扫描记录。
- **防碰扫描**：支持从“最近设计结果”选轨迹，或上传一条轨迹；结果高亮风险段，一键生成报告。

## 四、各功能流程

### 4.1 轨道设计（融防碰）流程

```
用户进入「轨迹设计」
    → 步骤1：输入靶点 (E, N, D)、井口 (E, N, D)
    → 步骤2：从基础数据选择邻井（或上传邻井 Excel），系统带出井口坐标列表
    → 步骤3：选择优化算法（PSO / B2OPT / GA-optiGAN），可选调整：种群/迭代、安全半径、入靶阈值等
    → 用户点击「开始设计」
    → 后端：校验输入 → 调用统一入口 optimize_well_trajectory(..., optimizer_fn, optimizer_kwargs)
    → 后端：返回 best_solution_dict、final_deviation、optimization_time、轨迹点集等
    → 前端：展示 12 参数表、3D 轨迹图（含邻井）、入靶偏差、耗时
    → 用户可：保存为设计井、导出报告/图片、跳转「防碰扫描」对该轨迹做扫描
```

**系统侧流程**：  
输入校验 → 加载邻井障碍物(create_multiple_well_obstacles) → 调用所选算法 → 结果转统一结构(best_solution_dict, final_deviation, 轨迹) → 写入设计记录(可选) → 返回前端。

### 4.2 基础数据与录井管理流程

- **井与井口**：新增/编辑井 → 填写井号、井名、井口(E,N,D)等 → 保存至数据库。列表支持按区块/井号筛选、导出 Excel。
- **邻井轨迹**：上传 Excel（与现有 41-37YH3.xlsx 等格式兼容）→ 解析并关联井号/井口 → 存文件路径+元数据；设计/防碰时按井勾选。
- **录井管理**：上传录井文件(如 CSV/Excel) → 关联井号、井深/时间 → 存储；查询按井+时间范围；列表+简单曲线展示。

### 4.3 防碰扫描流程

```
用户进入「防碰扫描」
    → 选择待扫描轨迹：从“已有设计”选一条 或 上传一条轨迹(如 Excel/表格)
    → 选择邻井（多选，从基础数据中的邻井列表来）
    → 输入安全半径（默认 10 m）、深度步长（默认 10 m）
    → 点击「执行扫描」
    → 后端：加载轨迹与邻井轨迹 → 调用防碰扫描引擎(如 _min_horizontal_distance_scan)
    → 返回：最小距离、最近点位置、风险等级(安全/预警/危险)、分段结果(可选)
    → 前端：表格+颜色标识风险段；可生成防碰扫描报告(PDF/Word)并导出
```

## 五、开发技术栈建议

### 5.1 前端

- **框架**：Vue 3 + TypeScript 或 React + TypeScript，适合表单多、数据表格多的中后台。
- **UI**：Element Plus / Ant Design Vue 等，表格、表单、步骤条、上传组件现成。
- **3D 轨迹图**：Three.js 或 ECharts 3D，与现有 Python 生成轨迹点对接（接口返回 E,N,D 点列即可）。
- **请求**：Axios，与后端 REST 通信；长时任务（如优化）建议轮询或 WebSocket 查状态。

### 5.2 后端

- **语言/框架**：Python 3.10+；Web 框架 FastAPI（异步、自动 OpenAPI，便于前后端协作）。
- **轨迹与优化**：沿用现有仓库：`minimal.py` 统一入口、`pso_well_trajectory`、`optimize_well_trajectory_b2opt`、`optimize_well_trajectory_ga_optigan`、`optimization_workflow`、`well_trajectory_objective` 等，作为“优化引擎”子模块封装成服务接口。
- **防碰扫描**：复用 `well_trajectory_objective` 中邻井障碍物与 `_min_horizontal_distance_scan` 等逻辑，封装为防碰服务。
- **任务队列**：Celery + Redis（或 FastAPI 后台线程）处理耗时优化任务，接口返回任务 ID，前端轮询结果。

### 5.3 数据与存储

- **数据库**：PostgreSQL 或 MySQL — 存井、井口、设计记录、录井元数据、用户/权限等。
- **文件存储**：邻井 Excel、上传轨迹、生成的轨迹图与报告 — 本地目录或 MinIO/OSS，数据库存路径。
- **配置与日志**：环境变量 + 配置文件；日志按模块划分，便于排查优化与防碰问题。

### 5.4 部署与运行

- **容器**：Docker + Docker Compose（前端 Nginx、后端 Uvicorn、Redis、PostgreSQL）。
- **进程**：Web 进程（FastAPI）、Worker 进程（Celery 执行优化与扫描），便于水平扩展。

## 六、可集成算法与扩展方式

### 6.1 当前已对接七段式统一接口的算法（可直接集成）

| 算法 | 入口函数 | 说明 |
|------|----------|------|
| **PSO** | `optimize_seven_segment_well_trajectory_pso` | 七段式权重目标，稳定可用，适合默认推荐 |
| **B2OPT** | `optimize_well_trajectory_b2opt` | 需 checkpoint 与 MLROpt 环境，适合有 GPU 资源时选用 |
| **GA-optiGAN** | `optimize_well_trajectory_ga_optigan` | 需 GA-optiGAN 子模块，适合研究/对比 |

统一调用方式（与 `minimal.py` 一致）：

```python
from minimal import optimize_well_trajectory
from pso_well_trajectory import optimize_seven_segment_well_trajectory_pso
from optimize_well_trajectory_ga_optigan import optimize_well_trajectory_ga_optigan
from optimize_well_trajectory_b2opt import optimize_well_trajectory_b2opt

result = optimize_well_trajectory(
    target_e=502.64, target_n=2790.71, target_d=2636.06,
    wellhead_position=(222, 2030, 0),
    excel_files=["41-37YH3.xlsx", "41-37YH5.xlsx"],
    wellhead_positions=[(208, 2015, 0.0), (209, 2000, 0.0)],
    optimizer_fn=optimize_seven_segment_well_trajectory_pso,  # 可换为上述三者之一
    optimizer_kwargs={...},
)
# result["best_solution_dict"] 为 12 参数，供可视化和报告
```

### 6.2 仓库内其它算法（需按七段式统一接口适配后可集成）

| 算法 | 文件/入口 | 集成建议 |
|------|-----------|----------|
| **SA（模拟退火）** | `optimize_well_trajectory_sa` | 改为使用 `create_objective_function(..., seven_segment_weighted)` 与 `get_seven_segment_parameter_bounds()`，返回 `best_solution_dict` / `final_deviation` |
| **CMA-ES** | `optimize_well_trajectory_cmaes` | 同上，七段式目标+统一返回格式 |
| **L-SHADE** | `optimize_well_trajectory_lshade` 或 L_SHADE 目录 | 同上 |
| **DE/GA** | `de_well_trajectory.optimize_well_trajectory_de` / `optimize_well_trajectory_ga` | 同上 |

适配要点：  
- 目标函数：`create_objective_function(config, well_obstacles, objective_mode="seven_segment_weighted", objective_weights={...})`。  
- 边界：`config.get_seven_segment_parameter_bounds()`。  
- 返回：至少包含 `best_solution_dict`（12 参数字典）、`final_deviation`、`best_fitness`、`optimization_time`。

### 6.3 系统内算法调度建议

- 前端下拉选择：「PSO」「B2OPT」「GA-optiGAN」；后端维护算法名到 `(optimizer_fn, default_kwargs)` 的映射。
- 新算法接入：实现上述统一接口后，在配置或代码中注册一行即可在界面中可选。

## 七、实施建议与阶段划分

1. **Phase 1 — 核心闭环**  
   - 后端：FastAPI + 轨迹设计接口（仅 PSO）+ 防碰扫描接口；数据存 JSON/SQLite 或 PostgreSQL。  
   - 前端：单页「轨迹设计」+「防碰扫描」；轨迹 3D 用 ECharts 或静态图链接。  
   - 目标：完成“输入靶点与井口 → 选邻井 → 跑 PSO → 看结果与 3D → 防碰扫描”闭环。

2. **Phase 2 — 数据与多算法**  
   - 基础数据管理（井、井口、邻井 Excel）入库与选择联动。  
   - 算法扩展：B2OPT、GA-optiGAN 接入同一接口并在前端可选。  
   - 录井管理：上传、关联井、列表查询。

3. **Phase 3 — 体验与交付**  
   - 设计/防碰报告模板与导出（PDF/Word/Excel）。  
   - 任务队列与异步计算，避免长时优化阻塞请求。  
   - 部署与权限：Docker 化、简单角色（如 设计员/审核员）。
